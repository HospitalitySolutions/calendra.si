Implemented: move staging/production Postgres passwords out of Docker Compose and .env files

Changed files:
- docker-compose.prod.yml
- docker-compose.staging.yml
- .env.example
- .env.staging.example
- scripts/docker-compose-with-aws-secrets.sh
- docs/STAGING_ENVIRONMENT.md
- docs/POSTGRES_SECRETS_MANAGER.md

What changed:
1. Production docker-compose no longer hard-codes POSTGRES_PASSWORD=calendra.
   It now requires POSTGRES_PASSWORD to be loaded from AWS Secrets Manager before compose starts.

2. Staging docker-compose no longer has a fallback POSTGRES_PASSWORD=change-me-staging.
   It now also requires POSTGRES_PASSWORD from AWS Secrets Manager.

3. Staging backend no longer receives SPRING_DATASOURCE_URL, SPRING_DATASOURCE_USERNAME, or
   SPRING_DATASOURCE_PASSWORD from docker-compose. Spring Boot loads these from AWS Secrets Manager
   through the existing application-staging.yml secret import.

4. Added scripts/docker-compose-with-aws-secrets.sh.
   Usage:
     scripts/docker-compose-with-aws-secrets.sh staging up -d --build
     scripts/docker-compose-with-aws-secrets.sh production up -d --build

   The script reads the selected AWS Secrets Manager JSON, exports POSTGRES_PASSWORD only for the
   docker compose process, then runs docker compose with the staging or production compose file.

5. The script accepts either POSTGRES_PASSWORD or SPRING_DATASOURCE_PASSWORD in the secret JSON.
   POSTGRES_PASSWORD is preferred. SPRING_DATASOURCE_PASSWORD is used as fallback.

6. Updated env examples and docs so POSTGRES_PASSWORD is not stored in .env / .env.staging.

Required AWS Secrets Manager keys:
- POSTGRES_PASSWORD
- SPRING_DATASOURCE_URL
- SPRING_DATASOURCE_USERNAME
- SPRING_DATASOURCE_PASSWORD

For compose-managed Postgres, POSTGRES_PASSWORD and SPRING_DATASOURCE_PASSWORD should be the same value.
